\section{Purpose}

Before launching 86+ batch training runs, we decoupled the
evaluation step from training to maximize GPU throughput and
confirmed the post-training pipeline produces correct and complete
artifacts. This report documents:

\begin{enumerate}[leftmargin=2em, itemsep=0.3em]
  \item The decoupled training/assessment architecture.
  \item Configuration changes applied.
  \item Fixes to \texttt{reeval\_checkpoints.py}.
  \item Colab T4 validation results confirming correctness.
  \item Example outputs (learning curves, score tables).
\end{enumerate}


\section{Architecture}

\subsection{Problem}

Running assessment episodes inline during training pauses gradient
computation to run 30 episodes per checkpoint. With 10 checkpoints,
this adds 300 assessment episodes per run---20--100\% overhead
depending on episode length. For Rainbow (2 gradient updates per
environment step), this overhead is proportionally more costly
relative to training time.

\subsection{Solution}

\textbf{Training phase} (A100, maximum throughput):
\begin{itemize}[leftmargin=2em, itemsep=0.1em]
  \item Assessment disabled during training
        (\texttt{eval\_every: 999999})
  \item 10 checkpoints saved at uniform intervals
    (\texttt{save\_every: 40000}, \texttt{keep\_last\_n: 10})
  \item No video recording during training
\end{itemize}

\textbf{Assessment phase} (T4 or CPU, cheap):
\begin{itemize}[leftmargin=2em, itemsep=0.1em]
  \item \texttt{reeval\_checkpoints.py} loads each checkpoint,
    runs 30 episodes, saves all artifacts
  \item Produces: \texttt{evaluations.csv}, \texttt{evaluations.jsonl},
    \texttt{per\_episode\_returns.jsonl}, \texttt{detailed/*.json}
  \item Optional video recording (\texttt{--record-video}) for
    selected runs only
\end{itemize}

\textbf{Two-tier assessment:} Video recording adds $\sim$3$\times$
overhead (frame rendering on every step). Fast assessment without
video runs first for all runs. Selective video recording runs
second on chosen checkpoints for thesis figures.

\section{Script Fixes}

Six issues were identified and fixed in
\texttt{scripts/reeval\_checkpoints.py}:

\medskip
\textbf{1. Missing sticky actions} (critical). The
\texttt{make\_atari\_env()} call did not pass
\texttt{repeat\_action\_probability}. Atari-100K training uses
0.25 (Machado et al.\ 2018), but assessment defaulted to 0.0.
Scores would have been incomparable. Fix: read the value from
\texttt{config.yaml} and pass it through.

\medskip
\textbf{2. NoisyNets mode}. Rainbow with NoisyNets
should use greedy assessment (\texttt{eval\_epsilon=0.0}) with
noise disabled via \texttt{model.eval()}. The script previously
used \texttt{eval\_epsilon=0.05} for all models. Fix: detect
Rainbow+NoisyNets from config and set epsilon accordingly.

\medskip
\textbf{3. No video recording}. Added \texttt{--record-video}
flag that records best-episode MP4 per checkpoint, matching the
inline format.

\medskip
\textbf{4. Missing per-episode data}. The script only wrote
\texttt{evaluations.csv}. Added output of
\texttt{per\_episode\_returns.jsonl} and
\texttt{detailed/eval\_step\_*.json} to match inline output.

\medskip
\textbf{5. Hardcoded CPU}. Added \texttt{--device} flag supporting
\texttt{cpu} and \texttt{cuda} for T4 inference.

\medskip
\textbf{6. Missing render mode}. Video recording requires
\texttt{render\_mode="rgb\_array"} passed to the environment.
Without it, \texttt{env.render()} returns \texttt{None} and no
frames are captured.
